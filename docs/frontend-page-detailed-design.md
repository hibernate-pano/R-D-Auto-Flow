## 2. 设计目标

前端工作台必须解决四个问题：

1. 用户如何手动指定一个 Jira Ticket 启动 Flow
2. 用户如何快速知道一个 Flow 运行到哪个阶段
3. 用户如何看到失败原因和关键日志
4. 用户如何在必要时执行人工干预

MVP 阶段不追求复杂视觉表达，重点是信息密度、状态清晰度和操作可控性。

MVP 前端包含以下页面与弹层：

1. Flow 列表页

3. 手动启动弹窗
4. 人工干预弹窗
5. 日志详情抽屉
6. 冲突提示弹窗

## 4. 全局信息架构

## 5. Flow 列表页

## 5.1 页面目标

让用户在进入系统后的第一屏就能完成以下动作：

1. 查看所有 Flow 的运行状态
2. 快速定位失败或阻塞的 Flow
3. 手动启动新的 Ticket Flow
4. 进入某个 Flow 查看详情

## 5.2 页面布局

建议布局为上下结构：

1. 顶部操作区
2. 中部筛选区
3. 主体列表区
4. 底部分页区

## 5.3 顶部操作区

包含以下元素：

1. 页面标题：Flow 工作台
2. “手动启动”主按钮
3. 全局搜索框
4. 刷新按钮

### 交互说明

1. 点击“手动启动”打开手动启动弹窗
2. 搜索框支持输入：
   - Jira Key
   - Ticket 标题关键字
   - Flow ID

## 5.4 筛选区

建议包含以下筛选项：

1. 状态筛选
   - 运行中
   - 等待人工处理
   - 已暂停
   - 已失败
   - 已完成
   - 已终止

2. 阶段筛选
   - 获取 Jira 内容
   - 解析源文档
   - 生成分析页
   - 创建分支
   - 开发中
   - 测试中
   - 回写中

3. 启动方式筛选
   - 手动启动
   - 重跑
   - 失败恢复

4. 时间筛选
   - 今日
   - 最近 7 天
   - 最近 30 天

## 5.5 列表字段

建议列表字段如下：

1. Jira Key
2. Ticket 标题
3. 当前阶段
4. 当前状态
5. 启动方式
6. 最近更新时间
7. 是否需要人工处理
8. 关联对象
9. 快捷操作

### 字段说明

#### Jira Key

- 可点击
- 点击进入详情页

#### Ticket 标题

- 单行截断
- hover 显示完整标题

#### 当前阶段

- 使用短标签显示
- 建议附带阶段序号

#### 当前状态

- 颜色区分：
  - 运行中：蓝色
  - 等待人工处理：橙色
  - 已失败：红色
  - 已完成：绿色
  - 已暂停：灰色
  - 已终止：深灰色

#### 是否需要人工处理

- 布尔标签
- `Yes` 时应显著提醒

#### 关联对象

- Jira
- Confluence
- GitHub

点击后在新标签页打开

#### 快捷操作

- 查看详情
- 重试
- 暂停
- 继续
- 终止

## 5.6 空状态

当列表为空时，显示：

1. “当前没有 Flow”
2. “手动启动一个 Jira Ticket”按钮

## 5.7 异常状态

当列表加载失败时，显示：

1. 加载失败提示
2. 重试按钮

## 6. 手动启动弹窗

## 6.1 使用场景

用户通过输入或搜索选择 Jira Ticket，发起新的 Flow。

## 6.2 弹窗字段

1. Jira Key
2. 启动模式
3. Repo Override
4. 备注
5. Source Flow（重跑 / 恢复时）

## 6.3 字段定义

### Jira Key

- 类型：支持搜索建议的文本输入
- 必填
- 占位文案：例如 `RD-1234`

校验：

1. 必填
2. 格式必须符合 Jira Key 约束
3. 输入时可调用 Jira 搜索接口展示候选
4. 失焦或提交前必须触发预检查

### 启动模式

- 类型：单选
- 默认值：首次执行

选项：

1. 首次执行
2. 重跑
3. 失败恢复

### Source Flow

- 当启动模式为“重跑”或“失败恢复”时显示
- 用于选择来源 Flow
- 恢复模式下可追加选择恢复阶段

### Repo Override

- 类型：可选文本输入或下拉
- 默认隐藏在“高级选项”中

### 备注

- 类型：多行文本
- 可选

## 6.4 按钮

1. 取消
2. 启动

### 启动按钮行为

1. 前端本地校验
2. 调用后端启动接口
3. 若启动成功：
   - 关闭弹窗
   - 刷新列表
   - 跳转详情页或 toast 提示
4. 若启动失败：
   - 在弹窗内显示错误

## 6.5 冲突场景

若该 Jira Ticket 已存在运行中的 Flow：

1. 不允许静默重复创建
2. 打开“冲突提示弹窗”
3. 提供以下选项：
   - 取消
   - 进入现有 Flow
   - 明确以重跑模式创建
   - 明确以恢复模式继续

## 7. Flow 详情页

## 7.1 页面目标

这是 MVP 的核心页面。用户需要在此页面完成判断、排障、干预、跳转。

## 7.2 页面结构

建议采用上下分区：

1. 顶部摘要区
2. 阶段时间线区
3. 关联对象区
4. Evidence 区
5. 日志区
6. 人工操作区

## 7.3 顶部摘要区

显示以下信息：

1. Jira Key
2. Ticket 标题
3. 当前状态
4. 当前阶段
5. 启动方式
6. 开始时间
7. 最近更新时间
8. 总耗时
9. 阻塞原因摘要

操作按钮：

1. 打开 Jira
2. 打开 Confluence 分析页
3. 打开 GitHub 分支

## 7.4 阶段时间线区

### 展示方式

建议使用垂直时间线。

### 每个阶段展示信息

1. 阶段名称
2. 阶段状态
3. 开始时间
4. 结束时间
5. 耗时
6. 重试次数
7. 是否人工推进
8. 错误摘要

### 阶段状态

1. 未开始
2. 进行中
3. 成功
4. 失败
5. 已跳过
6. 等待人工处理

### 阶段点击行为

点击某阶段后：

1. 滚动日志区到对应阶段
2. 高亮该阶段相关日志

## 7.5 关联对象区

显示以下对象：

1. Jira Ticket 链接
2. 源 Confluence 页面列表
3. 目标 Confluence 分析页
4. GitHub Repo
5. GitHub Branch

每个对象都应支持：

1. 复制链接
2. 新标签页打开

## 7.6 日志区

### 区块目标

让用户快速理解“哪里错了”和“为什么错了”。

### 日志工具栏

1. 关键字搜索
2. 阶段筛选
3. 日志级别筛选
4. 仅看错误开关
5. 刷新按钮

### 日志列表字段

1. 时间
2. 阶段
3. 级别
4. 事件类型
5. 摘要消息

### 日志行交互

点击某条日志，打开日志详情抽屉。

## 7.6A Evidence 区

### 区块目标

让用户清楚知道闭环是否具备推进条件，而不是只看日志猜测。

### 最低展示内容

1. 实现证据摘要
2. 测试执行证据摘要
3. 人工验证证据摘要
4. 审批结果摘要
5. 当前关闭门禁缺口

### 必备操作

1. 新增实现说明
2. 新增测试执行记录
3. 新增人工验证记录
4. 查看全部 evidence 明细

### 推进规则

1. 从 `implementation_waiting` 进入 `verification_waiting` 前，必须存在至少一条 `implementation_note`
2. 从 `verification_waiting` 进入 `verification_approval_waiting` 前，必须存在 `test_execution` 或 `manual_verification`

## 7.7 人工操作区

### 展示条件

在以下情况显著展示：

1. 当前 Flow 处于阻塞态
2. 当前 Flow 处于失败态
3. 当前 Flow 处于等待人工确认态

### 可用操作

1. 暂停
2. 继续
3. 重试当前阶段
4. 跳过当前阶段
5. 终止
6. 指定 Repo
7. 修正 Confluence 链接
8. 确认分析页通过
9. 要求补充分析内容
10. 确认验证通过
11. 要求补充验证内容

### 按钮行为要求

1. 所有动作都需要记录操作人
2. 破坏性动作必须二次确认
3. 成功后刷新详情页
4. 失败时展示错误信息
5. 按钮显示应以后端 `/available-actions` 返回为准

## 10. 冲突提示弹窗

## 10.1 使用场景

当用户手动启动一个 Jira Ticket，但系统发现它已有运行中的 Flow 或已有未清理的历史执行记录。

## 10.2 展示内容

1. 冲突 Ticket Key
2. 当前运行中的 Flow ID
3. 当前状态
4. 当前阶段
5. 最近更新时间
6. 系统建议的下一步触发模式

## 10.3 可执行动作

1. 取消
2. 查看现有 Flow
3. 明确以重跑模式继续
4. 明确以恢复模式继续

## 11. 前端状态管理

## 11.1 页面状态

前端至少需要管理以下状态：

1. 列表加载状态
2. 详情加载状态
3. 启动弹窗提交状态
4. 人工操作提交状态
5. 日志筛选状态
6. evidence 列表加载状态
7. evidence 录入提交状态

## 11.2 推荐原则

1. 查询状态与表单状态分离
2. 错误状态不要静默吞掉
3. 用户操作后优先乐观刷新摘要，再拉全量详情

## 13. 页面级验收标准

## 13.1 Flow 列表页

1. 用户可以看到 Flow 列表
2. 用户可以按状态、阶段、Jira Key 筛选
3. 用户可以从列表快速进入详情
4. 用户可以从列表直接发起手动启动

## 13.2 手动启动弹窗

1. 输入合法 Jira Key 后可正常提交
2. 非法 Jira Key 有明确错误提示
3. 冲突 Ticket 不会无提示重复启动

## 13.3 Flow 详情页

1. 能展示完整阶段轨迹
2. 能展示关联对象
3. 能查看日志
4. 能执行人工操作

## 13.4 日志详情抽屉

1. 能展示日志详细内容
2. 敏感信息脱敏
3. 能快速关闭并回到原上下文

2. 更细的操作权限展示
3. 更复杂的审批态 UI

---

## 15. 总结

MVP 前端的核心不是“页面多”，而是“关键动作都能闭环”。

如果这几件事能稳定完成，前端就达标：

1. 用户能手动指定 Ticket 启动
2. 用户能看到 Flow 跑到哪一步
3. 用户能查到失败原因
4. 用户能执行人工干预
